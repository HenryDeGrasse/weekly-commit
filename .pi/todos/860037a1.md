# Reports & Analytics — Metrics Enhancement Plan

## Backend Endpoints Already Built but NOT Rendered on the Frontend

These 4 endpoints exist in `ReportController` and `reportApi.ts` with full types — they just need UI cards:

### 1. 🏰 Chess Distribution (`GET /api/reports/chess-distribution`)
- **Data:** `ChessDistributionReportEntry` — `{ teamId, weekStart, distribution: Record<string, number> }`
- **Shows:** How many commits per chess piece (KING/QUEEN/ROOK/BISHOP/KNIGHT/PAWN) for a given week
- **Recommended viz:** Stacked horizontal bar or pie/donut per week, or a single-week breakdown with week selector
- **Value:** Reveals if the team is over-indexing on low-impact work (too many pawns) or has too many critical items (kings)

### 2. 🔀 Scope Change Volume (`GET /api/reports/scope-changes`)
- **Data:** `ScopeChangeReportEntry` — `{ userId, weekStart, scopeChangeCount }`
- **Shows:** Number of scope changes per user per week
- **Recommended viz:** Bar chart by week (aggregate), optionally drill-down per person. Show trend line.
- **Value:** High scope volatility = planning instability. Surfaces teams that churn post-lock.

### 3. 🌲 RCDO Coverage (`GET /api/reports/rcdo-coverage`)
- **Data:** `RcdoCoverageReportEntry` — `{ rcdoNodeId, weekStart, plannedPoints, achievedPoints, commitCount, teamContributionBreakdown }`
- **Shows:** Planned vs achieved effort mapped to RCDO tree nodes + which teams contribute
- **Recommended viz:** Tree-map or grouped bar by RCDO node. Needs an RCDO node selector.
- **Value:** Shows which strategic objectives are getting effort and which are starving.

### 4. ⏰ Exception Aging (`GET /api/reports/exception-aging`)
- **Data:** `ExceptionAgingEntry` — `{ exceptionId, teamId, userId, exceptionType, severity, weekStartDate, createdAt, ageInHours }`
- **Shows:** Unresolved manager exceptions with how old they are
- **Recommended viz:** Table sorted by age (oldest first), color-coded by severity. Badge for exception type.
- **Value:** Accountability — stale exceptions mean managers aren't reviewing. SLA tracking.

## NEW Metrics to Add (require new backend + frontend work)

### 5. 📊 Velocity Trend (Achieved Points / Week)
- **Source:** Already in `team_week_rollup.total_achieved_points`
- **Calculation:** Plot `totalAchievedPoints` as a line chart over the selected weeks
- **Also show:** Rolling 4-week average overlay
- **Value:** The single most important health metric — is the team delivering more or less over time?

### 6. 🎯 Achievement Rate % (Achieved / Planned per week)
- **Source:** Already in `team_week_rollup` — `totalAchievedPoints / totalPlannedPoints * 100`
- **Recommended viz:** Line or area chart with a target line (e.g., 80%)
- **Value:** Are people planning realistically? Consistently low = over-commitment; consistently high = sand-bagging.

### 7. 👤 Individual Contributor Leaderboard / Heatmap
- **Source:** `user_week_fact` — already has per-user `totalPlannedPoints`, `totalAchievedPoints`, `commitCount`, `carryForwardCount`
- **Recommended viz:** Heatmap grid (users × weeks), color intensity = achievement rate. Or a ranked table.
- **Value:** Spot outliers — who's consistently over-committing, who's killing it, who might need help.

### 8. 🔄 Carry-Forward Streak Distribution
- **Source:** `weekly_commit.carry_forward_streak` — exists on the commit entity
- **New endpoint needed:** Aggregate streaks across team: how many commits have streak 0, 1, 2, 3+
- **Recommended viz:** Histogram or segmented bar
- **Value:** Repeated carry-forwards (streak ≥ 2) are exceptions. This shows how chronic the problem is.

### 9. 📈 Capacity Utilization
- **Source:** `user_week_fact.total_planned_points` vs. configured capacity from `OrgConfig` / `TeamConfigOverride` / `CapacityOverride`
- **New endpoint needed:** Compare planned points to effective capacity budget per user, aggregate to team
- **Recommended viz:** Gauge or bar per week showing % capacity used (planned / budget)
- **Value:** Under-utilization = waste. Over-utilization = burnout risk. Ideally 80-95%.

### 10. 🏷️ Outcome Breakdown (by CommitOutcome)
- **Source:** `weekly_commit.outcome` — ACHIEVED, PARTIALLY_ACHIEVED, NOT_ACHIEVED, CANCELED
- **New endpoint needed:** Count commits per outcome per week for a team
- **Recommended viz:** Stacked bar chart per week (4 segments). Shows ratio.
- **Value:** More granular than points-based PvA. "50% of my commits were PARTIALLY_ACHIEVED" is actionable.

### 11. ⚠️ Exception Frequency by Type
- **Source:** `manager_review_exception.exception_type` — MISSED_LOCK, AUTO_LOCKED, MISSED_RECONCILE, OVER_BUDGET, etc.
- **New endpoint needed:** Count exceptions by type over the date range
- **Recommended viz:** Grouped bar or heatmap (type × week)
- **Value:** Diagnose systemic issues — if AUTO_LOCKED is high, people aren't locking. If OVER_BUDGET is high, capacity config is wrong.

### 12. 🔗 Ticket Linkage Rate
- **Source:** `weekly_commit.work_item_id` — how many commits have a linked ticket
- **New endpoint needed:** `linked_count / total_count` per week
- **Recommended viz:** % line chart per week
- **Value:** Traceability metric — are people connecting their weekly commits to tickets/work items?
