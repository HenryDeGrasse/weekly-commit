I found two real plan gaps that would make the implementation path unreliable in this codebase.

- `AiSuggestion` only has `userId`, `planId`, `commitId`, `suggestionType`, `prompt`, `suggestionPayload`, etc. — no `teamId` or `weekStartDate`. The plan’s team-insight retrieval step (`findBySuggestionTypeAndCreatedAtAfter(...)` then “filter later by team context in payload”) is not workable as written, because step 7 never says to persist team/week context in a retrievable form for each `TEAM_INSIGHT`. The endpoint `GET /api/teams/{id}/week/{weekStart}/ai-insights` needs a deterministic way to fetch the right team+week insights.
- The plan adds `generatePersonalInsights(planId)` and a read endpoint for `PERSONAL_INSIGHT`, but nothing ever calls that method. In the current service layout, `/api/plans/{id}/ai-insights` would otherwise just read an empty table subset forever. The plan needs an explicit generation trigger for personal insights.

- Update `OpenRouterAiProvider.loadPromptTemplate()` for the new suggestion types, otherwise the new prompt files won’t be used.
