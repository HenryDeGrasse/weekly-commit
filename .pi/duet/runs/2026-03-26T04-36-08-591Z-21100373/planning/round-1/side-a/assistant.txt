## Plan Summary

I explored the codebase thoroughly and produced a 13-step implementation plan. Here's my reasoning:

### Key Observations
- **Backend**: Spring Boot 3.4 with Java 21, uses `java.net.http.HttpClient` for external API calls (established in `OpenRouterAiProvider`), strategy pattern via `AiProviderRegistry`, all AI features degrade gracefully
- **Frontend**: React 18 + TypeScript, custom `useQuery` hook pattern, `createXxxApi(client, actorUserId)` factory pattern for API clients, existing `AiFeedbackButtons` component for reuse
- **Testing**: JUnit 5 + Mockito + AssertJ on backend (H2 in-memory, `ai.provider: stub`), Vitest + Testing Library on frontend with vi.mock patterns
- **No `@EnableAsync` exists yet** — needed for the `@Async` indexing calls

### Step Ordering Rationale
1. **Steps 1-4** (foundation): Config, EmbeddingService, PineconeClient, ChunkBuilder — each independently testable, no cross-dependencies
2. **Step 5** (SemanticIndexService + hooks): Depends on steps 2-4, modifies existing service constructors with optional `@Autowired(required=false)` to avoid breaking existing tests
3. **Steps 6-7** (query + insight services): The RAG read path, depends on Pinecone + Embedding
4. **Step 8** (controller endpoints + DTOs): Backend API layer, depends on steps 6-7
5. **Steps 9-12** (frontend): API client → individual components → wiring, each independently verifiable
6. **Step 13** (integration wiring): Wire components into existing pages, update existing test mocks

### Key Design Decisions
- Constructor injection with `@Autowired(required=false)` for SemanticIndexService in existing services to avoid breaking 472 existing backend tests
- Followed established patterns exactly (HttpClient usage, API factory pattern, graceful degradation, AiContext type constants, StubAiProvider canned responses)
- The namespace strategy uses org ID (looked up via team → org chain) for Pinecone multi-tenancy
