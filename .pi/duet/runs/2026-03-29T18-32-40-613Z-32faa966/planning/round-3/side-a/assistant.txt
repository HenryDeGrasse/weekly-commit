The plan has been updated. Here's a summary of the fix:

## What changed

**The core problem:** Step 12's original design persisted new `AiSuggestion` rows on every GET request, meaning `suggestionId` values changed on every page load/refetch. Step 13's localStorage-based dismissal by `suggestionId` would break immediately.

**The fix — separate read from write, following the existing `RiskDetectionService` pattern:**

**Step 12 (backend):**
- Split into **two public methods**:
  - `getRecommendations(planId)` — **read-only**. Queries `AiSuggestionRepository.findByPlanIdAndSuggestionType(planId, "PLAN_RECOMMENDATION")`, deserializes persisted rows, returns stable `suggestionId` values. No new rows created. Safe to call repeatedly.
  - `generateAndPersistRecommendations(planId, userId)` — **delete-then-regenerate** (same pattern as `RiskDetectionService.detectAndStoreRiskSignals`). Deletes existing `PLAN_RECOMMENDATION` rows, computes fresh recommendations, persists new rows.
- Split into **two endpoints**:
  - `GET /api/ai/plans/{planId}/recommendations` — calls `getRecommendations()` (read-only, stable IDs)
  - `POST /api/ai/plans/{planId}/recommendations/refresh` — calls `generateAndPersistRecommendations()` (regenerate)
- Added test cases for stable-read behavior and stale-row deletion

**Step 13 (frontend):**
- Uses the GET endpoint for display (stable IDs for dismissal + feedback)
- Adds a "Refresh recommendations" button that calls the POST refresh endpoint
