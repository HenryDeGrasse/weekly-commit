## Key decisions addressing the gap analysis issues:

### 1. Step 6 — WebFlux/Reactor not available → Switched to `SseEmitter`
The backend has `spring-boot-starter-web` only (no `webflux`/`reactor-core`). I replaced the `Flux<ServerSentEvent<...>>` design with Spring MVC's native `SseEmitter` (from `org.springframework.web.servlet.mvc.method.annotation`), which requires **zero new dependencies**. The implementation uses an `ExecutorService` or `@Async` to run the query in a background thread and emit SSE events via `emitter.send()`.

### 2. Step 8 — `UserWeekFact` doesn't have per-chess-piece data → Use `WeeklyCommit` directly
`UserWeekFact` only stores aggregate `totalPlannedPoints/totalAchievedPoints` plus `kingCount/queenCount` counts. It does **not** store per-chess-piece achievement rates or per-piece estimates. The plan now explicitly:
- Uses `UserWeekFactRepository` for **aggregate** stats (overall achievement rate, carry-forward probability)
- Uses `WeeklyCommitRepository` with a new `@Query` (JOIN to `weekly_plan` for date filtering) for **per-chess-piece** achievement rates and average estimates
- The `WeeklyCommit` entity has `chessPiece`, `estimatePoints`, and `outcome` fields which provide all needed data
