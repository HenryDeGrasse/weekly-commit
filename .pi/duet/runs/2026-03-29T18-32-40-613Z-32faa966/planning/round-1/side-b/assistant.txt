The plan is broadly workable and the step ordering is sensible. I only found one blocker that should be fixed before implementation.

Non-blocking suggestions:
- In Step 7, make sure the streaming fetch path includes the auth token from the host bridge, since the rest of the frontend API layer consistently sends `Authorization`.
- In Step 8, consider mirroring the existing privacy pattern used by risk signals/manager AI endpoints for the calibration endpoint if it is meant to expose user-specific performance history.
- In Step 12, clarify the “lowest-priority commit” rule a bit: `priorityOrder` is user ordering, not chess-piece priority, so it would help to specify whether to prefer the last ordered commit, a PAWN if present, or some combined heuristic.

Verdict: changes_requested
Blocking issues:
- Steps 12 and 13 are missing the identifier needed to make recommendations actionable in the existing frontend architecture. `PlanRecommendation`/`PlanRecommendationResponse` do not include any stable `recommendationId`, `riskSignalId`, or `suggestionId`, but Step 13 requires per-recommendation dismissal by ID and `AiFeedbackButtons`, which currently post feedback against a `suggestionId`. As written, the frontend step cannot fully implement its own deliverables without extra backend changes outside the plan.
