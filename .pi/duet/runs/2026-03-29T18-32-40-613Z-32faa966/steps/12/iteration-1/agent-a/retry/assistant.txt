## What Was Implemented

### New files created:

**`backend/src/main/java/com/weeklycommit/ai/service/PlanRecommendationService.java`**
- `generateAndPersistRecommendations(UUID planId, UUID userId)`: delete-then-regenerate pipeline
  - Deletes stale `PLAN_RECOMMENDATION` AiSuggestion rows
  - Fetches risk signals; triggers `detectAndStoreRiskSignalsById` if none stored
  - For OVERCOMMIT/UNDERCOMMIT: runs `WhatIfService.simulate` with `REMOVE_COMMIT` on the lowest-priority commit (PAWN preferred)
  - For REPEATED_CARRY_FORWARD: runs `WhatIfService.simulate` with `MODIFY_COMMIT` (halved estimate) on the carry-forward commit
  - Persists each via `AiSuggestionService.storeSuggestion("PLAN_RECOMMENDATION", ...)` to get stable `suggestionId`s
  - Graceful degradation: if WhatIfService or CalibrationService fails, recommendation is still persisted (partial, `whatIfResult = null`)
- `getRecommendations(UUID planId)`: read-only, deserializes payload from persisted rows, stable IDs, no regeneration
- `PlanRecommendation` public nested record with all required fields

**`backend/src/main/java/com/weeklycommit/ai/dto/PlanRecommendationResponse.java`**
- API response DTO with `UUID suggestionId` field, factory `of()` / `ofList()` methods

**`backend/src/test/java/com/weeklycommit/ai/service/PlanRecommendationServiceTest.java`**
- 6 tests covering all specified scenarios: OVERCOMMIT recommendation, no-risks-empty, WhatIfService failure → partial recommendation, multiple risks → multiple recommendations, stable IDs from persisted rows, delete-stale-rows-first

### Files modified:

**`AiController.java`**: Added `PlanRecommendationService` injection, `GET /api/ai/plans/{planId}/recommendations`, and `POST /api/ai/plans/{planId}/recommendations/refresh` endpoints (both require `X-Actor-User-Id` header)

**`AiControllerTest.java`**: Added `@MockBean PlanRecommendationService planRecommendationService` (required for `@WebMvcTest` to wire the controller)

### Design note
`WhatIfResponse` contains `Map<UUID, Integer>` which would cause UUID key deserialization issues if stored in JSONB and read back. Instead, only the key narrative fields (riskType, description, suggestedAction, narrative, confidence) are stored in `suggestionPayload`. The `whatIfResult` is populated on fresh generation but `null` on retrieval — which is the expected behavior per the step description (frontend calls the refresh endpoint when it needs fresh simulation data).
