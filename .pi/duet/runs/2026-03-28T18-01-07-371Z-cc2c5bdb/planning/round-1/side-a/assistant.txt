**Summary of reasoning:**

The plan converts the 6-step markdown plan into the PlanDraft JSON schema, preserving the original sequencing and intent. Key decisions:

1. **Steps 1-2 use `build` check only** — these modify only backend test files, and `build` runs `./gradlew build -x test` which compiles everything including test code. The `unit` check would fail because `./gradlew test` excludes `@Tag("eval")` tests and wouldn't validate the eval-specific changes.

2. **Step 3 uses `build` check** — creating two new JSON files at the repo root doesn't need lint/typecheck (those run frontend/shared workspace linting, not JSON validation). `build` is sufficient to confirm nothing is broken.

3. **Steps 4-5 use `build` check** — the Node.js script is standalone and doesn't participate in the TypeScript typecheck or ESLint workflows (those are scoped to `frontend` and `@weekly-commit/shared` workspaces). `build` confirms the overall project still compiles.

4. **Step 6 uses `unit` + `build`** — this is where the Node test runner tests get wired into the root `test` script, so `unit` (which runs `npm test`) will execute them. `build` provides the full end-to-end verification.

5. **Descriptions are detailed** with critical implementation notes from the plan (e.g., the `callJudge` HTTP approach since `OpenRouterAiProvider.generateSuggestion()` can't be used for judge calls, CommonJS requirement, the 6-arg constructor change needed for `EvalResult`).
