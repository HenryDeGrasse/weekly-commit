## Plan Summary: Weekly Commit Module v1 — 20 Steps

### Architecture Decision
- **Polyglot monorepo**: TypeScript frontend (Vite/React) + Java 21 backend (Spring Boot/Gradle) + SQL (Flyway/PostgreSQL)
- Root `package.json` orchestrates all npm-based checks by shelling out to both frontend and backend toolchains

### Step Sequencing Rationale

**Steps 1–2 (Foundation):** Scaffold the monorepo and define the complete domain model + database schema upfront. This ensures all subsequent backend and frontend steps have shared types and a stable schema to build against.

**Steps 3–8 (Backend API, bottom-up by domain):** Build backend services in dependency order:
- **3**: RCDO hierarchy first (needed by commit validation)
- **4**: Plan/Commit CRUD (core domain)
- **5**: Lock + snapshots (lifecycle critical path)
- **6**: Scope changes + reconciliation (post-lock lifecycle)
- **7**: Carry-forward + tickets (remaining domain objects)
- **8**: Team view + manager review + permissions (aggregation layer on top of all domains)

**Steps 9–14 (Frontend, matching backend surfaces):** Build UI for the 5 PRD surfaces, starting with routing scaffold, then RCDO, My Week, Lock/Reconcile, Team Week, and Tickets/Search.

**Steps 15–17 (Cross-cutting: Notifications + AI):** These are additive features that don't block the core workflow. AI comes after core is stable, matching the PRD's "AI must degrade gracefully" requirement.

**Steps 18–20 (Reporting, Config, Audit):** Derived read models, admin configuration, audit logging, accessibility, and final integration polish. These wrap up the system.

### Key Design Choices
- All 20 acceptance criteria from PRD §29 are covered across the steps
- AI steps (16–17) come after all core backend/frontend, per PRD §30's risk analysis about over-scoping
- Every step requires all 4 checks (lint, typecheck, unit, build) to ensure the project stays green
- Backend builds first, then frontend, enabling end-to-end integration naturally
